# 算法可视化网站质量目标（Skill 基线）

## 1. 质量维度

本 Skill 的产出质量以 8 个维度评估，满分 100：

| 维度 | 分值 | 说明 |
| --- | --- | --- |
| 教学有效性 | 20 | 用户能否快速理解算法步骤和状态迁移 |
| 可视化质量 | 20 | 画布信息密度、布局稳定、动画清晰度 |
| 代码同步质量 | 15 | 分镜与代码高亮、变量旁注一致性 |
| 交互体验 | 10 | 输入、控制、快捷键、进度条拖拽的易用性 |
| 工程质量 | 10 | 模块化、可维护性、无明显反模式 |
| 可靠性 | 10 | 异常回退、缓存策略、空数据安全 |
| 性能与稳定性 | 10 | 大规模数据下可交互、不卡顿 |
| 发布可用性 | 5 | CI 通过、Pages 可访问 |

## 2. 评分门槛

- A 级（>= 90）：可直接作为对外演示版本
- B 级（80-89）：可内部发布，建议做 1 轮优化
- C 级（70-79）：仅开发验证，不建议外发
- D 级（< 70）：不通过

## 3. 一票否决项（任一命中直接不通过）

1. 出现紫色主题或紫色强调色
2. 分镜与代码高亮明显错位
3. 关键交互不可用（进度条不可拖拽 / 快捷键失效）
4. 输入非法数据导致页面崩溃
5. lint/type-check/build 任一失败
6. GitHub Pages 流水线配置缺失或错误

## 4. 最低交付定义（Definition of Done）

满足以下条件才可标记 Done：

1. 功能实现：符合 SKILL.md 中硬约束
2. 测试通过：本地与 Playwright 验收通过
3. 工程通过：lint/type-check/build 全绿
4. 文档齐全：README + TEST_PLAN + 验收报告
5. 证据完整：输出命令、截图、结果记录

## 5. 质量回归策略

- 每新增一个可视化能力，都要补至少 1 条 e2e 用例
- 每次重构都要回归运行核心验收矩阵
- 发现错位问题（代码行与画步不同步）优先修复，不可延期
