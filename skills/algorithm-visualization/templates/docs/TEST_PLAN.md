# 测试计划

## 1. 目标

验证算法可视化网站在功能、交互、稳定性和教学效果方面达到可发布标准，并可依据统一门禁判定 PASS/FAIL。

## 2. 测试分层

### 2.1 单元测试（Vitest）

范围：`features/`、`services/`、`components/`（纯逻辑部分）

必测项：
- `usePlayback`：播放状态、步进边界、seek 范围、reset 行为
- `solutionRegistry`：步骤结构、输入校验、随机生成、多语言代码存在性
- `validateInput`：每种解法的非法输入拒绝 + 合法输入解析

运行命令：`npm run test:unit`
通过标准：exit code = 0

### 2.2 E2E 测试（Playwright）

- E2E-001: 首屏单屏布局与关键元素可见
- E2E-002: 顶部外链（题目/Hot100/GitHub）新标签行为
- E2E-003: 输入合法性（非法不重算、合法重算）
- E2E-004: 样例与随机输入
- E2E-005: 语言切换与刷新记忆
- E2E-006: 代码高亮与变量旁注同步
- E2E-007: 快捷键 Left/Right/Space/R
- E2E-008: 进度条拖拽跳步
- E2E-009: 画布缩放与平移
- E2E-010: 算法思路弹窗
- E2E-011: 多解法切换隔离（有多解法时）
- E2E-012: 交流群浮层与二维码比例
- E2E-013: 速度偏好刷新后保留
- E2E-014: GitHub 徽标与 Star 数展示
- E2E-015: 无紫色主题 token

### 2.3 工程测试

- qa:guard
- lint
- type-check
- build
- test:unit
- test:e2e
- GitHub Action

### 2.4 回归测试

以下修改触发全量回归：

1. 画布渲染结构变更
2. 分镜绑定机制变更
3. 输入规则变更
4. 控制交互变更

## 3. 测试环境

- Node.js 20+
- Chrome / Chromium
- 本地高位端口（30000-65535）

## 4. 退出标准

1. P0/P1 缺陷为 0
2. 核心 e2e 用例通过率 100%
3. 质量门禁命令全部通过
4. AC-001~AC-022 全部通过（仅 AC-006 可在无多解法时 N/A）
5. 最终判定模型 B1-B7 全部满足

## 5. 失败分流与回归策略（强制）

当任一 Gate 失败时，必须进入以下流程：

1. 失败分流：环境问题 / 脚本问题 / 产品缺陷
2. 缺陷分级：P0/P1/P2/P3
3. 修复策略：最小变更优先，避免扩大影响面
4. 回归策略：先受影响集，再核心必过集合
5. 复测策略：flaky 用例最多重试 2 次，仍失败按缺陷处理
6. 报告更新：记录本轮失败项、修复动作、复测结果

- 单元测试失败必须优先修复，才能进入 E2E 测试
- 新增 feature 未补单元测试视为 P1 缺陷

未完成上述流程，不得给出 PASS 结论。
